# 事件补写 SOP

本文档规定每次补写事件时的操作流程。目标是让事件库能长期扩展到上万条，同时保持时代准确、触发稳定、效果可控、文件可维护。

## 1. 开始前确认批次

每次只写一个明确批次，建议 50-150 条。

批次说明至少包含：

```text
批次：daily-school-001
目标：补 80 条学校日常
年龄：6-22
年份：1900-2035
重点条件：时代、城乡、性别、家境、教育压力、考试特质
涉及文件：src/data/events/daily/school.js 或新拆文件
```

如果批次跨度太大，先拆小。例如“补改革开放事件”应拆成“包产到户”“个体户”“下海”“沿海务工”“下岗”等。

## 2. 先读现有上下文

写事件前必须先看这些内容：

1. 目标事件文件。
2. `src/data/events.js` 聚合入口。
3. `src/data/tagLabels.js`，确认新标签是否需要展示名。
4. 引擎支持的条件和效果：`src/engine/conditions.js`、`src/engine/effects.js`。
5. 如果涉及人生状态字段，查看 `src/engine/state.js`。

不要凭记忆添加不存在的字段、条件或效果。

## 3. 决定放在哪个文件

优先按语义放入已有文件：

- 基础生命周期：`src/data/events/core/`
- 历史大时代：`src/data/events/history/`
- 日常生活：`src/data/events/daily/`
- 后续职业、地区、家庭等如果拆细，按 `src/data/events/README.md` 的规则新增目录。

当单文件超过约 100-150 条事件时继续拆细。新增文件后必须更新 `src/data/events.js` 的 import 和聚合数组。

## 4. 判断“一个事件”还是“多个事件”

默认一个人生语义事件只保留一个事件 ID。

使用 `text` variants 的情况：

- 只是口吻不同。
- 同一件事在男女、城乡、地区、阶层中的表达不同。
- 同一结果下，因为性格、标签、时代不同而换文案。

拆成多个事件 ID 的情况：

- 触发年份或年龄明显不同。
- 后果不同。
- 自动结果及后果明显不同，且现有 `outcomes` 不能通过条件和权重表达。
- 优先级、冷却、最大次数不同。
- 历史意义不同，例如“恢复高考”和“普通高考”。
- 需要被其他事件单独依赖或阻断。

## 5. 先定触发边界

每条事件先写清楚边界，再写文案：

- `ageRange`：人生年龄。
- `yearRange`：发生年份。
- `birthYearRange`：出生代际。
- `stage`：人生阶段。
- `genders`：只在事件确实和性别强相关时使用。
- `birthFamilyClasses`：出生家庭阶层。
- `familyTags` / `blockFamilyTags`：家庭、城乡、户籍、代际背景。
- `birthRegions` / `currentRegions`：出生地或当前所在地。
- `requiresEvents` / `blocksEvents`：事件链条依赖。
- `conditions`：更细的状态判断。

年代边界优先于现代直觉。1930 年代不要出现投简历、公司 offer、互联网；1960 年代不要出现市场化职业选择；1990 年前后要区分单位、个体户、下海、沿海工厂。

历史事件必须做细粒度触发约束，不能只按一个粗时代段放开。写每条事件时至少逐项判断：

- 具体年份：事件是持续多年，还是只在某几年高概率出现。
- 具体地区：全国性、区域性、城市性、农村性、前线、后方、沦陷区、根据地、侨乡、沿海、内陆。
- 具体人群：年龄、性别、户籍、职业、家庭成分、是否学生、是否工人、是否农民、是否已有相关经历。
- 具体状态：是否已经参军、是否逃难、是否在工厂、是否有孩子、是否受过教育、是否有迁移经历。
- 具体概率：不是“可能发生”就高权重；罕见或地区性经历应低权重，并用 `weightModifiers` 收紧。

例子：

- 不允许把 `1900-1949` 作为一个大段，然后让所有人都有概率遇到日军轰炸。
- 抗战轰炸应限制在 `1937-1945`，并进一步限制到可能遭遇空袭的城市、交通节点、战时后方或相关迁移状态。
- 长征、湘江、苏区、支前、拉壮丁、偷渡过港、三线建设、国企下岗、WTO 外贸工厂，都必须有各自的年份、地区、人群或前置状态约束。
- 如果不能确定某事件的地区和时间边界，先查资料或把事件降级为更普遍的日常/传闻，不要写成亲历。

## 6. 性别处理原则

性别要考虑，但不要硬塞。

适合用 `genders` 限定：

- 事件生理上只适用于某一性别。
- 当时制度、家庭分工或社会期待导致路径明显不同。
- 选择和后果对不同性别完全不同。

适合用 `text` variants 或 `weightModifiers`：

- 同一事件男女都可能发生，但体验、压力、旁人评价不同。
- 家庭期待略有差异，但最终效果相同。

不需要处理性别：

- 天灾、物价、普通健康、很多学习和工作日常。

## 7. 文案写法

文案要短、有画面、能体现时代。

建议：

- 每条主文案 1-3 句。
- 避免说明书口吻。
- 历史事件可以有情绪表达，但不要每条都喊口号。
- 日常事件要低烈度，能填满普通年份。
- 同一事件可放多个 `text` variants，制造重复游玩变化。

避免：

- 现代词穿越到旧时代。
- 把复杂历史写成单一立场口号。
- 每条事件都过于传奇，导致人生没有日常呼吸。
- 大段解释背景，事件不是百科。

## 8. 效果设计

效果要服务人生状态，不要为了“有变化”乱改。

常用效果：

- 属性：`attrs.intelligence`、`attrs.health`、`attrs.charisma`、`attrs.resilience` 等。
- 资源：财富、声望、压力、关系、教育、职业等。
- 标签：`addTag` / `removeTag`。
- 特质：`addTrait` / `removeTrait`。
- 计数器：经历次数、长期压力、家庭负担等。
- 事件链：`scheduleEvent`、`setFlag`、`addTimedModifier`。
- 终局：`die`、`triggerEnding`。

数值原则：

- 日常小事通常是 -2 到 +3。
- 关键人生转折可以到 -8 到 +12。
- 历史灾难、战争、重大疾病可更大，但要谨慎。
- 同一事件不要同时改太多无关字段。
- 有长期意义的结果优先加 tag，而不是一次性大幅加数值。

## 9. 自动结果设计

人生推进阶段禁止使用 `choices`，也禁止新增任何需要玩家点击选项才能继续的交互。事件存在多种可能走向时使用 `outcomes`，由引擎自动决定。

适合使用自动结果：

- 升学、辍学、参军、迁移、结婚、创业、转行。
- 面对风险时的不同策略。
- 家庭与个人目标冲突。

不适合使用多个结果：

- 出生背景。
- 被动遭遇。
- 纯日常小事。
- 历史洪流中个人没有实际选择空间的事件。

自动结果需要配置 `resultText` 和 `effects`；可使用 `conditions`、`baseWeight`、`weightModifiers` 让属性、特质和经历影响结果。所有结果都应在同一年度推进中完成，不能让 UI 进入等待状态。

校验脚本会拒绝任何带 `choices` 字段的事件，这是不可回退的核心交互约束。

## 10. 权重与频率

使用这些字段控制出现频率：

- `baseWeight`：基础权重。
- `priority`：高优先级事件更容易在候选中胜出。
- `cooldown`：同类体验不要连续刷。
- `maxOccurrences`：人生中最多出现几次。
- `weightModifiers`：按性别、地区、家境、标签、时代调整概率。

建议：

- 大时代事件低频但有辨识度。
- 日常事件中频，用于填普通年份。
- 结局事件高优先级但严格条件。
- 同一语义的反复事件加 cooldown 或 maxOccurrences。

## 11. 标签与特质

新增 tag 或 trait 前先搜索是否已有近义项。

需要展示给玩家的 tag 必须在 `src/data/tagLabels.js` 补 label。内部纯控制标记如果不会展示，也要尽量命名清楚。

命名建议：

- 使用英文 snake_case。
- 语义具体，不要过泛。
- 路径型标签用 `_path`，记忆型标签用 `_memory`，时代型标签用 `_era` 或具体历史名。

示例：

- `factory_worker_path`
- `neighbor_help_memory`
- `sent_down_youth`
- `ai_tool_user`

## 12. 历史准确性

涉及明确历史事件时，要确认年份和语境。

需要特别小心：

- 红军革命、苏区、长征、湘江。
- 抗日战争、沦陷区、后方、游击队。
- 解放战争、支前、拉壮丁。
- 新中国成立、土改、合作化、公社。
- 大跃进、困难时期。
- 文化大革命、知青、停课、工农兵大学生。
- 改革开放、恢复高考、包产到户、个体户、下海。
- 偷渡过港、沿海务工、国企下岗。
- WTO、北京奥运、移动互联网、疫情、AI 时代。

如果年份、政策、地域差异不确定，先查资料再写；不要把模糊记忆写进事件条件。

## 13. 地区、城乡与阶层

同一历史事件在不同人身上的意义不同。

写事件时至少考虑：

- 城市、农村、县城、沿海、内陆、边疆、东北、港澳侨乡。
- 家庭成分、干部/工人/农民/商人/知识分子。
- 户籍、单位、集体、市场、平台。
- 家中是否有老人、孩子、兄弟姐妹、病人。

不要把全国经验写成单一城市经验，也不要把城市白领经验默认成所有人经验。

## 14. 日常事件标准

日常事件的价值是让空白年份有生活。

日常事件应覆盖：

- 童年玩耍、家务、挨训、零食、邻里。
- 学校作业、同桌、老师、考试、小冲突。
- 家庭饭桌、亲戚、过年、借钱、照顾老人。
- 工作迟到、加班、饭局、被夸、被骂、转岗。
- 健康小病、睡眠、牙疼、运动、体检。
- 老年散步、老友、孙辈、丧偶、回忆。

日常事件不一定要强效果，甚至可以只给很小的情绪、关系或压力变化。

## 15. 验证流程

每批写完必须验证。

基础语法：

```bash
node --check src/data/events.js
```

导入验证：

```bash
node --input-type=module -e "import('./src/data/events.js').then(({ events }) => console.log(events.length))"
```

结构验证建议至少检查：

- 事件 ID 是否重复。
- `requiresEvents` / `blocksEvents` 是否引用不存在的事件。
- 新 tag 是否缺少展示 label。
- 每个事件是否有 `id`、`title`、`category`、`text`。
- `ageRange` / `yearRange` 是否符合常识。

模拟验证：

- 至少跑一批随机人生。
- 观察是否存在大量空白年份。
- 观察旧时代是否刷出现代事件。
- 观察某些高权重事件是否刷屏。
- 抽几条完整人生读叙事是否顺。

## 16. Review 输出格式

每批完成后给出：

```text
批次：daily-school-001
新增事件：80
修改事件：6
新增标签：12
涉及文件：...
重点覆盖：...
自查结果：无重复 ID / 无缺失引用 / 无缺失标签 / 模拟通过
需要 review 的点：...
```

如果有历史资料不确定，必须列出来，不能假装确定。

## 17. 不要做的事

- 不要把事件直接写进 `src/data/events.js`。
- 不要为了数量复制同义事件。
- 不要把表达差异拆成大量独立事件。
- 不要新增没有引擎支持的字段。
- 不要把现代语言放进旧时代。
- 不要一次改太多系统逻辑和事件数据。
- 不要删除用户未要求删除的其他改动。
