# 内容质量批量评测

`scripts/evaluate_content.js` 是事件库的可重复质量门禁。它既扫描数据定义，也可读取 `batch_simulate.js` 产生的 summary、years、events 三张 CSV，把“是否贴合时代、地点、阶级和人物状态”尽量转成可追踪的数字与样本。

评测器不会改变游戏逻辑，也不会引入任何中途选择。

## 推荐工作流

先跨全部出生年代跑至少 1000 局：

```bash
node scripts/batch_simulate.js \
  --count 1200 \
  --seed CONTENT-QA \
  --max-age 100 \
  --out reports/content-qa.csv

node scripts/evaluate_content.js \
  --summary reports/content-qa.csv \
  --report reports/content-qa.evaluation.md \
  --json reports/content-qa.evaluation.json
```

`--summary` 给出 summary 路径后，评测器会自动寻找同名前缀的 `.years.csv` 和 `.events.csv`。也可以显式指定：

```bash
node scripts/evaluate_content.js \
  --summary reports/content-qa.csv \
  --years reports/content-qa.years.csv \
  --events reports/content-qa.events.csv
```

不提供 CSV 时，只扫描事件定义和固定年谱：

```bash
node scripts/evaluate_content.js
```

需要把分数作为 CI 门禁时使用：

```bash
node scripts/evaluate_content.js \
  --summary reports/content-qa.csv \
  --strict \
  --min-score 80
```

`--strict` 会在出现阻断问题或低于最低分时返回非零状态；默认模式只报告，不会因内容告警中断后续分析。`--samples` 可控制每类问题最多打印多少个样本，默认 8。

## 固定年谱专项回放

固定年谱不是普通事件之间穿插关键节点，而是一部从出生到死亡逐年写定的作品。使用中性 id 强制回放：

```bash
node scripts/batch_simulate.js \
  --count 2 \
  --historical-life chronicle_hunan_1893_a \
  --seed CHRONICLE-QA \
  --out reports/chronicle-qa.csv

node scripts/evaluate_content.js --summary reports/chronicle-qa.csv
```

评测器会核对：

- 年谱从精确出生年开始，年份连续且没有缺口。
- 每年恰好一条固定事件，没有普通事件或自然流变混入。
- 最后一年明确结束生命。
- 不包含 `choices`、随机 `outcomes` 或玩家分叉。
- 两个及以上 seed 的标题、正文、类别和结局完全一致。
- 玩家可见字段不泄露 `hiddenTerms`。
- CSV 只写 `chronicle_id`，不导出研究层的 `sourceSubject`。

## 评测维度

### 数据与条件一致性

- event id 的依赖、阻塞、调度引用是否存在。
- 年龄、自然年、出生年、人生阶段是否存在可达交集。
- 地区、城市层级、户口、家庭阶级、性别过滤值是否合法。
- CSV 中真实触发记录是否违反直接时间、出生地区、迁移后所在地、阶级或性别条件。
- `year = birth_year + age`、summary 计数和三表关联是否一致。

所在地校验会按事件顺序重放迁移效果，使用“事件触发前”的地点判断，避免拿当年结束后的地点误报迁移事件。

### 连续性、覆盖与重复

- 无事件年份比例，以及连续三年以上空年的样本。
- 事件定义覆盖率、未出现事件清单、分类分布和分年代覆盖。
- 高频前十事件占比，防止少量万能事件淹没时代内容。
- 同一人生重复触发同一事件的额外次数。
- 同一人生重复出现完全相同最终文案的额外次数。

重复事件不一定是逻辑错误，例如感冒、年夜饭可以再次发生；但若每次文案完全相同，长篇人生会明显露出“抽卡池”的机械感。优先给高频日常事件增加年代、地点、阶级和人物状态变体，或设置合适的 cooldown / maxOccurrences。

当前正式引擎每年只取一条主叙事；`daily_` 事件若未显式声明 `maxOccurrences`，默认一生只叙述一次。真实生活当然会反复感冒、争吵和排队，但文学时间线没有必要用同一段话反复记账。确实需要复现的事件应显式声明次数，并提供足够多的条件化文案变体；文本选择器会优先使用本局尚未展示过的变体。

“具体事件占比高”不等于人生具有结构。感冒、通勤、饭桌和邻里切片即使各有独立 id，连续出现时仍会让作品显得只是日常事件清单。分层批测因此另算 `narrative_tier`：`turning_point`、`historical_pressure` 和 `consequence` 合计为结构事件，`texture` 为日常纹理；同时检查每局成年后的结构密度与最长连续纹理年。评测时应通读连续段落，确认后果确实承接先前状态，而不是只追求一个好看的比例。

当具体事件池暂时没有更合适的候选时，`life_quiet_year` 不再反复抽取一小组固定句子，而是按自然年、当前城乡位置、年龄阶段、健康、财富、教育、职业和家庭状态组合一段微型年景。组合器会避开本局已经出现过的同构文案；它是连续叙事的兜底，不应取代继续扩充可考据的具体事件。

### 人生结局

- 婴儿、18 岁前、40 岁前结束率。
- 各出生年代平均终年和早逝率。
- ending id 与 death reason 分布，已死亡但未分类的结局。

某个年代的早逝率显著异常时，应先检查高优先级死亡事件。当前选择器中，最高优先级候选会被选中；`baseWeight` 很低并不会把一个孤立的最高优先级死亡事件变成“小概率”。

### 文案与时代感

- 过短正文、过长正文、不同条目完全重复文案。
- 实际触发年份早于现代词汇建议起点的样本，例如手机、互联网、短视频、微信、高铁。
- 带笑意、打趣、尴尬、小聪明等生活纹理信号的比例，作为“沉重历史中是否有呼吸感”的弱指标。

现代词表是启发式告警，不是历史裁决。一个词出现在解释未来的句子里可能合理；但玩家正在阅读的是当年叙述，仍建议优先使用当时人物能够自然说出或感受到的表达。

## 如何读分数

分数用于比较修改前后，不应取代人工文学审读：

- 92–100：优秀，仍需抽样通读完整人生。
- 82–91：良好，有局部密度或措辞问题。
- 70–81：可用但需优化。
- 55–69：内容风险较高。
- 0–54：不建议发布，先处理阻断问题和系统性重复。

建议每轮遵循同一闭环：固定 seed 跑基线 → 保存 JSON 指标 → 修复最高频问题 → 用相同 seed 重跑 → 比较覆盖、重复、冲突、早逝和分年代指标。新增大量事件后再换 seed 复验，防止只对一组随机样本过拟合。

## 人工审读仍然必要

程序能发现“1930 年出现手机”之类硬冲突，却不能可靠判断一段文字是否真正深刻、有余味或幽默得恰到好处。每次发布前至少人工通读以下样本：

- 晚清乡村贫农、商埠店员、士绅女性各一局。
- 民国战争地区与非战争地区各一局。
- 1950–1977 年工人、农民、干部/知识分子家庭各一局。
- 改革年代国企职工、个体户、外出务工家庭各一局。
- 当代城乡、不同财富层级和性别各一局。
- 每一部固定年谱从出生到结局完整通读两遍。

审读时逐年问四个问题：这个人在当时当地真的可能遇见它吗；它与前一年积累的状态一致吗；删去时代名词后是否仍像一段人的生活；连续读二十年是否出现同一种句式和同一种感伤。
